Key Findings

  1. Worker isolation works

    OpenCode correctly ran in /tmp/worker-a without escaping the sandbox. File system boundaries held under timeout pressure.

  2. Skill auto-loading works

    The worker-orchestrator skill was discovered and evaluated by the spawned OpenCode agents. Skill path resolution is reliable.

  3. Prompt-file pattern is correct

    No shell-quoting failures observed. Piping prompts via stdin file avoids the fragility of positional arguments with multi-line content.

  4. Provider rate-limiting is the critical failure mode

    When API limits hit, workers hang indefinitely. There is no graceful degradation — the stream errors silently until timeout kills the worker.

  5. Monitoring gap

    execute_code + subprocess.Popen gives no live visibility. We only see results after timeout or kill. Real-time heartbeat streaming is needed.

Recommendations

Provider circuit-breaker

If 429 detected, respawn on alternate provider immediately. Do not wait for timeout.

Aggressive timeout tuning

Shorten OpenCode run timeout to 60–90s when API limits are known tight. Fail fast, recover faster.

Pre-warm quota checks

Probe provider quota before spawning batches. Avoid launching workers into known blocked providers.

Local-model fallback

Consider llama.cpp/ollama worker tasks to bypass external API limits entirely for appropriate workloads.
