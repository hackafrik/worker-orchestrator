Observations

Key Findings

  1. 01

    Worker isolation works. OpenCode correctly ran in /tmp/worker-a without escaping the sandbox.

  2. 02

    Skill auto-loading works. The worker-orchestrator skill was discovered and loaded from ~/.agents/skills/worker-orchestrator/.

  3. 03

    Prompt-file pattern is correct. No shell-quoting failures observed when delivering prompts via file rather than positional arguments.

  4. 04

    Provider rate-limiting is the critical failure mode. When API limits hit, workers hang indefinitely with no graceful degradation. The usage_limit_reached error surfaced 1300s before reset.

  5. 05

    Monitoring gap. execute_code + subprocess.Popen gives no live visibility — results only appear after timeout or process exit.

Action Items

Recommendations

  • Add a provider circuit-breaker. If 429 detected, respawn on alternate provider immediately. Do not wait for the reset timer.

  • Shorten OpenCode timeout. Set run timeout to 60-90s when API limits are known tight. Fail fast, respawn fast.

  • Pre-warm provider quota check. Before spawning batches, verify provider quota to avoid predictable rate-limit failures.

  • Consider local-model fallback. Use llama.cpp or ollama for worker tasks to avoid external API limits entirely.

Methodology

How We Test

All evaluation data is generated from live orchestration runs, not mocked or simulated. Workers are real OS processes. Failures are real provider errors. The reports live in the repository at eval/ and are committed alongside code changes.
