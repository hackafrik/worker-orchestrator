Universal Worker Orchestrator

A Hermes Agent skill that turns your AI agent into a fleet commander across heterogeneous worker pools. Spawn CLI agents, HTTP API endpoints, Python scripts, and Docker containers — then monitor, evaluate, rebalance, and synthesize their outputs.

Architecture

Hermes acts as the orchestrator that spawns, monitors, evaluates, and synthesizes output from external AI workers across four categories:

CategoryExamplesBest For
CLIOpenCode, Codex, Claude Code, Aider, GooseFull-agent coding tasks
HTTP_APIollama, vLLM, OpenAI, Anthropic, GroqRemote/local inference endpoints
PYTHON_SCRIPTCustom inference scripts, data pipelinesBespoke Python workers
DOCKERGPU inference containers, sandboxed agentsIsolated, reproducible runs

Worker Adapter Interface

Every adapter exposes four operations: spawn, monitor, kill, evaluate.

  • CLI (adapter_cli.py): subprocess.Popen with new session for clean tree-kill.
  • HTTP_API (adapter_http.py): REST/SSE/NDJSON with streaming consumer subprocess.
  • PYTHON_SCRIPT (adapter_python.py): Thin wrapper around Python script subprocess.
  • DOCKER (adapter_docker.py): docker run lifecycle via docker CLI inspect.

State Machine

Validated lifecycle transitions prevent ambiguous states:

PENDING → SPAWNING → RUNNING → MONITORING → EVALUATING
                                    ↓
                        ┌───────────┼───────────┐
                        ↓           ↓           ↓
                     SUCCEEDED    FAILED      KILLED
                                    ↓
                                TIMED_OUT

Workflow

Step 1: Discover & Plan

python3 scripts/discover_workers.py > .worker-registry.json

Use the registry to pick available adapters, then decompose into 2–5 independent subtasks across phases.

Step 2: Prepare task_spec

Each subtask is a JSON task_spec:

{
  "task_id": "worker-a-phase-0",
  "phase": 0,
  "worker_category": "CLI",
  "adapter_name": "opencode-cli",
  "prompt": "Implement auth endpoints...",
  "workdir": "/tmp/worker-a",
  "timeout_seconds": 300,
  "rubric": {
    "expected_output_markers": ["login", "register"],
    "expected_files": ["auth.py"]
  },
  "runtime": {
    "command": ["opencode", "run", "--dir", "/tmp/worker-a"]
  }
}

Category-specific runtime blocks are documented in the Adapter Reference below.

Step 3: Spawn via Orchestrator

python3 scripts/run_orchestrator.py --task-spec task_spec.json

This runs the full lifecycle:

  1. State machine init → SPAWNING → RUNNING
  2. Monitor loop with timeout
  3. Evaluation against rubric
  4. Cost recording
  5. Terminal state (SUCCEEDED / FAILED / KILLED / TIMED_OUT)

Step 5: Evaluate

CriterionScaleDescription
PresencePass/FailDid worker produce output?
Correctness1–5Output matches subtask?
Completeness1–5All parts addressed?
Quality1–5Professional standard?

PASS if Presence=Pass AND average >= 3.0.

Step 6: Feedback & Respawn

For FAILED workers:

  1. Analyze which rubric criteria scored low
  2. Generate specific corrective feedback
  3. Respawn on same or alternate provider (respecting pool caps)

Circuit-breaker respawns (rate-limiting):

  • Use scripts/circuit_breaker.py to detect API exhaustion
  • If rate-limited, respawn immediately on alternate provider without quality feedback
  • Counts against max 2 respawns per subtask

Respawn prompt template:

You previously attempted: [original task]
Your output was insufficient because: [specific reason]
Please redo with these corrections: [specific instructions]

Step 8: Synthesize

python3 scripts/synthesize_outputs.py \
  --inputs /tmp/worker-a /tmp/worker-b \
  --output /tmp/final \
  --type code

Use --phase N --dependency-inputs /tmp/phase-N-1 to include prior phase outputs.

Adapter Reference

CLI Adapter

Spawns subprocess via subprocess.Popen in a new session for clean tree-kill.

  • Transport: pid, pgid, pty, invocation
  • Kill: os.killpg(SIGTERM) → grace period → SIGKILL
  • Artifacts: Any files in artifacts_dir excluding internal logs
  • Monitor: Checks ps -p to detect process exit; reads .exitcode sentinel if present

Runtime block:

{
  "command": ["opencode", "--acp", "--stdio"],
  "extra_args": ["--model", "gpt-4o"],
  "stdin_text": "{prompt}",
  "use_shell": false,
  "pty": false,
  "cwd": "."
}

HTTP API Adapter

Speaks to HTTP/S APIs (REST, SSE, NDJSON). Designed for ollama, vLLM, and any OpenAI-compatible endpoint.

  • Transport: request_id, stream_pid (background stream consumer), pgid
  • Spawn: Writes request.json for audit; for streaming protocols spawns a background consumer subprocess
  • Monitor: Checks if stream consumer process has exited; parses JSON usage metadata for tokens
  • Kill: SIGKILL on stream consumer process
  • Artifacts: Response body saved to stdout.log; events to events.jsonl

Runtime block:

{
  "base_url": "http://localhost:11434",
  "endpoint": "/api/generate",
  "method": "POST",
  "headers": {"Content-Type": "application/json"},
  "payload": {"model": "llama3.1", "stream": true},
  "stream_protocol": "sse"
}

Python Script Adapter

Subprocess wrapper for arbitrary Python scripts.

  • Transport: pid, pgid, invocation
  • Use case: Custom inference scripts, data processors, local model frontends
  • Args: Supports {task_spec_file} placeholder substitution

Runtime block:

{
  "script_path": "/path/to/worker.py",
  "interpreter": "python3",
  "args": ["--task-spec", "{task_spec_file}"],
  "stdin_text": "{prompt}"
}

Docker Adapter

Wraps docker run lifecycle.

  • Transport: container_name, docker_cmd, local_pid, pgid
  • Monitor: docker inspect to read container State.Status and ExitCode
  • Kill: docker stop -t 5 → docker kill
  • Mounting: runtime.mounts array with source, target, mode

Runtime block:

{
  "image": "ghcr.io/someorg/inference-worker:latest",
  "entrypoint": ["python3", "worker.py"],
  "command": ["--mode", "generate"],
  "mounts": [{"source": "/host/data", "target": "/data", "mode": "ro"}],
  "env": {"API_KEY": "***"},
  "network": "bridge",
  "user": "1000:1000"
}

Adding a New Adapter

  1. Create scripts/adapter_<category>.py
  2. Implement spawn, monitor, kill, evaluate
  3. Add to _resolve_adapter() in run_orchestrator.py
  4. Add to discover_workers.py detection logic
  5. Update documentation with transport schema and test notes

Safety & Limits

  • Max 6 active workers total
  • Max 2 respawns per subtask
  • Kill stuck workers after timeout
  • Clean up temp dirs after synthesis
  • Workers are sandboxed — copy required files into workdir before spawn
  • Codex requires git — git init in workdir when using Codex CLI
  • Always diff+merge config files when multiple workers modify the same file

Pitfalls & Lessons Learned

terminal(background=true) with opencode hangs indefinitely.
Use execute_code + subprocess.Popen with stdin piping instead.

Shell quoting with multi-line prompts is fragile.
Always pipe prompts via stdin file, never as positional shell arguments.

Multiple workers modifying the same config files will overwrite each other.
Diff and merge rather than blind copy.

Workers may create tests with wrong assumptions about function names/APIs.
Verify test imports against actual source exports after generation.

Worker sandboxing is strict.
OpenCode cannot read files outside its workdir. Copy required files before spawn.

Synthesis picks up node_modules noise.
Instruct workers to write findings to a SPECIFIC file and read only that file.

Provider API rate limits hang workers silently.
Set aggressive timeouts when quotas are tight. Use circuit-breaker to detect and swap providers.

Lean copied workdirs can break npm/TypeScript verification.
If compilation is required, copy node_modules too or run npm install in workdir. Otherwise do authoritative verification in the real repo after merging.
