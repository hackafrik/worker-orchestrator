Why orchestrate

One expensive model burns tokens fast. A fleet costs less.

Running everything on a single top-tier model is wasteful. The Universal Worker Orchestrator splits the load across an orchestrator plus a heterogeneous worker fleet — CLI agents for coding, HTTP endpoints for inference, Python scripts for data pipelines, and Docker containers for isolated GPU runs.

The orchestrator itself consumes only 15–25% of total cost. The rest goes to workers you choose based on task requirements and budget.

Worker categories

Heterogeneous pools. One unified interface.

Every adapter exposes the same four operations: spawn, monitor, kill, evaluate. Swap providers without rewriting orchestration logic.

Architecture

Phase-based orchestration with synthesized dependencies.

Tasks decompose into 2–5 independent subtasks across ordered phases. Later phases consume synthesized outputs from earlier phases.

Step 01

Discover & Plan

Probe the environment with discover_workers.py to build a registry of available adapters, binaries, and API endpoints.

Step 02

Spawn & Monitor

Launch workers via category-specific adapters. Monitor with ps -p, stream consumers, or docker inspect.

Step 03

Evaluate & Rebalance

Score outputs against rubrics. Respawn failed workers on alternate providers up to the global cap and retry limit.

synthesize outputs
Final

Merge & Deliver

synthesize_outputs.py merges worker artifacts into a unified deliverable. Use --phase N --dependency-inputs to chain multi-phase pipelines.

State machine

Validated lifecycle transitions.

Every worker moves through a formal state machine. No ambiguous states. No silent failures.

PENDING → SPAWNING → RUNNING → MONITORING → EVALUATING → SUCCEEDED / FAILED / KILLED / TIMED_OUT
