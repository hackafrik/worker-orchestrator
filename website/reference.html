Setup

Installation

Two ways to install. Both place the skill in ~/.hermes/skills/worker-orchestrator where Hermes auto-discovers it on startup.

bash 
# Option 1 — direct clone
git clone https://gitlab.com/hackafrik/worker-orchestrator.git ~/.hermes/skills/worker-orchestrator

# Option 2 — skills.sh ecosystem
npx skills add hackafrik/worker-orchestrator

# Verify
npx skills add hackafrik/worker-orchestrator -l

Requirements

  • Hermes Agent
  • OpenCode CLI installed and authenticated
  • Codex CLI installed and authenticated

Scope

When to Use

  • Complex tasks that can be split into independent subtasks
  • Tasks where cheaper models can do the heavy lifting
  • Projects requiring parallel workstreams (backend + frontend + docs)
  • Any time the user explicitly mentions workers, delegation, or parallel agents

When NOT to Use

  • Simple one-step tasks (read file, search web, run single command)
  • Tasks requiring tight real-time coordination between workers
  • Tasks where Hermes can complete faster than spawn overhead (~30s per worker)

Process

Workflow

Eight sequential steps. Each step feeds the next. The orchestrator loops on evaluate-respawn until all tasks pass or hit retry limits.

  1. 01

    Decompose

    Break the task into 2-5 independent subtasks. Each must be self-contained, produce a concrete artifact, and not depend on other workers unless you plan sequential phases.

    Example: "Build a FastAPI auth service" decomposes into: (A) auth endpoints, (B) pytest tests, (C) README.

  2. 02

    Prepare & Spawn

    Create isolated workdirs. Copy required project files. Init git for Codex. Spawn workers via execute_code (OpenCode) or terminal(pty=true) (Codex).

    Critical: Workers are sandboxed. They cannot read files outside their workdir.

  3. 03

    Monitor

    OpenCode monitoring is batch-style via proc.communicate(timeout=300). Codex on PTY can use process(action="poll").

    Look for progress markers (file reads, edits), error patterns (Error:, Exception:), and completion signals (Wrote to...).

  4. 04

    Evaluate

    Score every worker against a rubric. Pass threshold: Presence=Pass AND average >= 3.0.

    CriterionScaleDescription
    PresencePass/FailDid worker produce any output?
    Correctness1-5Does output match the subtask?
    Completeness1-5All parts addressed?
    Quality1-5Professional standard, no obvious errors?
  5. 05

    Respawn

    For FAILED workers, generate specific corrective feedback and respawn. Max 2 respawns per subtask. Circuit-breaker detects rate limits and respawns immediately on alternate provider.

  6. 06

    Dynamic Spawn

    After the first batch, spawn additional workers if gaps remain, synthesis is thin, or user feedback arrives mid-orchestration. Finished workers no longer count toward the 6-worker limit.

  7. 07

    Rebalance

    Shift work between OpenCode and Codex pools based on capacity, reliability, and task fit. This is a mixed fleet — not a one-way fallback chain.

  8. 08

    Synthesize

    Merge all worker outputs into a single coherent deliverable. Resolve conflicts, ignore node_modules noise, and credit which worker produced what.

Implementation

Spawning Workers

OpenCode — execute_code + subprocess

The reliable method. Avoids TUI hangs and shell-quoting issues.

python 
import subprocess, os, pathlib

prompt_path = pathlib.Path("/tmp/prompt.txt")
prompt_path.write_text("YOUR MULTI-LINE PROMPT HERE")

proc = subprocess.Popen(
    ["opencode", "run", "--dir", "/tmp/worker-a"],
    cwd="/tmp/worker-a",
    stdin=open(prompt_path, "r"),
    stdout=subprocess.PIPE,
    stderr=subprocess.STDOUT,
    text=True,
    env=os.environ.copy()
)
stdout, _ = proc.communicate(timeout=300)

Codex — terminal with PTY

Codex requires an interactive terminal.

bash 
terminal(command="codex exec 'YOUR_TASK_PROMPT_HERE'",
         workdir="/tmp/worker-b",
         pty=true)

Provider Allocation

PoolMax ActiveNotes
OpenCode3Spawned via execute_code. No live process() monitoring.
Codex3Spawned via terminal(pty=true). Can poll with process().
Global6Total across all providers.

Valid mixes: 3+3, 3+1, 2+3. Invalid: 4+2 (OpenCode cap exceeded), 3+4 (global cap exceeded).

Observation

Monitoring

OpenCode workers return all output at once when the subprocess exits. Parse stdout for errors and completion signals. Codex workers on PTY can be polled live.

Batch Monitoring (OpenCode)

python 
workers = [
    ("worker-a", ["opencode", "run", "prompt-a"], "/tmp/worker-a"),
    ("worker-b", ["opencode", "run", "prompt-b"], "/tmp/worker-b"),
]

for name, cmd, cwd in workers:
    proc = subprocess.Popen(cmd, cwd=cwd, stdout=subprocess.PIPE,
                            stderr=subprocess.STDOUT, text=True)
    stdout, _ = proc.communicate(timeout=300)
    print(f"=== {name} completed (exit {proc.returncode}) ===")
    print(stdout[-2000:] if len(stdout) > 2000 else stdout)

Live Monitoring (Codex)

bash 
process(action="poll", session_id="WORKER_SESSION_ID")
process(action="log", session_id="WORKER_SESSION_ID", limit=50)

Quality

Evaluation

Objective check first (file exists, not empty, expected keywords). Then Hermes reasoning for Correctness, Completeness, Quality.

Scoring

Average = (Correctness + Completeness + Quality) / 3. Pass if Presence=Pass AND average >= 3.0.

Circuit Breaker

Before evaluating quality, check if the failure was infrastructure (rate limit) rather than task quality.

python 
cb = subprocess.run(
    ["python3", "scripts/circuit_breaker.py", "/tmp/cb_check.txt"],
    capture_output=True, text=True
)
result = json.loads(cb.stdout)

if result["is_rate_limited"]:
    print(f"RATE LIMIT: {result['provider']} — {result['recommendation']}")
    # Respawn immediately on alternate provider

Severity rules: Critical (>5min reset) → alternate provider immediately. High (>1min) → alternate immediately. Medium (<1min) → 30s pause, retry once, then alternate.

Recovery

Respawn

Max 2 respawns per subtask (3 total attempts). After 3 failures, synthesize what you have or flag for user review.

Respawn Prompt Template

text 
You previously attempted: [original task]
Your output was insufficient because: [specific reason]
Please redo with these corrections: [specific instructions]

Circuit-Breaker Respawns

Rate-limit failures bypass quality analysis. Respawn the same task on the alternate provider immediately. Counts against the 2-respawn limit.

Scaling

Dynamic Spawn

The orchestrator does not stop after the first batch. Finished workers free up slots.

  • Gap filling: Worker found 5 bugs but only fixed 3 → spawn another for the rest.
  • Dependency resolution: Worker B needs a function Worker C was supposed to write → spawn after C finishes.
  • Quality boost: Synthesis is thin → spawn for docs, examples, edge cases.
  • User pivot: User adds requirements mid-flight → spawn to address.

Distribution

Provider Rebalancing

Do NOT treat Codex as a fallback. Both pools are first-class.

  • Start with the provider mix that best fits the phase.
  • If one provider is saturated, place the next subtask on the other if capacity exists.
  • Retry within the same provider for prompt-specific failures.
  • Reassign to the other provider for provider-specific failures or saturation.
  • Last resort: do the subtask directly in Hermes.

Examples: Review-heavy phase → 3 OpenCode + 2 Codex. Fix-heavy phase → 2 OpenCode + 3 Codex if Codex patches better.

Output

Synthesis

  1. Read terminal logs from all workers (primary source).
  2. Read files workers were instructed to create (secondary source).
  3. Ignore node_modules, vendor dirs, and auto-generated noise.
  4. Merge related outputs. Resolve conflicts.
  5. Produce final deliverable in the user's requested format.
  6. Report what each worker did, what failed, what was respawned.

Avoiding noise: Instruct workers to write findings to a specific file (e.g. review.md). Read only that file during synthesis.

Craft

Worker Prompt Engineering

Every worker prompt must be self-contained, atomic, constrained, and output-directed.

good 
Review the Python files in this directory. Check for bugs, security issues,
and performance problems. Write your findings to /tmp/worker-a/review.md
as a structured markdown report with HIGH/MEDIUM/LOW severity sections.
bad 
Review the code and tell me what you find.

Too vague — worker may stream findings in terminal instead of writing a file.

Lessons

Pitfalls & Lessons Learned

  • terminal(background=true) with opencode run hangs indefinitely

    The opencode TUI blocks on a pseudo-TTY that never receives input. Always spawn OpenCode via execute_code with Python subprocess.Popen.

  • Shell quoting with opencode run is fragile

    Complex prompts containing quotes, newlines, or backticks break shell parsing. Prefer stdin piping via subprocess.Popen(stdin=open(prompt_path)) for all non-trivial prompts.

  • Multiple workers overwrite the same config files

    If Worker A and Worker B both update package.json, the second copy overwrites the first. Always diff and merge config files rather than blind copy.

  • Workers may create tests with wrong assumptions

    A worker writing tests may assume function names that don't exist. After tests are created, verify imports against actual exports and fix assertions as needed.

  • Synthesis picks up node_modules noise

    When workers copy entire projects, synthesis ingests node_modules READMEs. Instruct workers to write findings to a specific file and read only that file.

  • Git init timing matters

    Git init on an empty workdir before copying files means git won't track the copies. Either copy first then git init && git add -A, or use diff original modified instead of git diff.

  • Lean workdirs break npm/TypeScript verification

    Excluding node_modules keeps copies fast, but tsc --noEmit or npm run lint may fail. Either copy dependencies too, or do authoritative verification in the real repo after merging.

  • Provider rate limits hang workers silently

    When OpenCode or Codex hits a 429, the process may block indefinitely. Set aggressive timeouts (60-90s) when limits are tight. If a worker times out with no output, assume provider exhaustion and respawn on alternate provider.

Walkthrough

Example Session

User: "I need a Python CLI tool that converts CSV to JSON. Have workers build the core converter, the CLI interface, and the tests. Then combine everything."

  1. Decompose: Worker A (core logic), Worker B (Click CLI), Worker C (pytest tests).
  2. Spawn: 3 OpenCode workers in parallel via execute_code + subprocess.Popen.
  3. Monitor: Capture stdout/stderr when each subprocess exits.
  4. Evaluate: All outputs pass rubric.
  5. Synthesize: Merge into single project at user-requested location.
  6. Report: "3 workers completed. Core logic + CLI + tests merged into ~/csv2json/"
