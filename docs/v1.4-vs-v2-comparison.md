# worker-orchestrator v1.4 vs v2 comparison

## Purpose

This document is the promotion gate for deciding whether v2 should eventually replace or partially back-port into v1.4.

v1.4 remains the stable baseline.
v2 is the experimental rewrite in a separate repo.

## Scope of comparison

Compared repos:
- v1.4 lineage: `/home/hackafrik/worker-orchestrator`
- v2 rewrite: `/home/hackafrik/worker-orchestrator-v2`

This comparison only uses evidence currently present in the repos and local evaluation runs.

## High-level summary

### v1.4 strengths
- Simpler and more focused on the original OpenCode/Codex worker model.
- Proven baseline for the original workflow.
- Less abstraction overhead.

### v2 strengths
- Generalized adapter architecture across CLI, HTTP API, Python script, and Docker workers.
- Formal worker state machine with explicit transition auditability.
- Cost ledger separated from worker execution.
- Clearer artifact/log/handle persistence model.
- Broader end-to-end coverage in local evals.

### Current recommendation
- Do not replace v1.4 yet.
- Continue validating v2.
- v2 now looks structurally better, but promotion should wait for more real-world parity evidence.

## Important findings from testing

The newer coverage and parity work exposed several real bugs in v2:
- `adapter_http.py` blocking mode originally returned a handle but never executed the request.
- `adapter_http.py` also incorrectly converted blocking non-200 responses into `SUCCEEDED` during monitor.
- `adapter_docker.py` did not persist container exit codes for short-lived `docker run --rm` jobs, so successful containers could be reported as failed after removal.
- `run_orchestrator.py` always passed `--model None` to `track_cost.py`, which broke orchestration runs when `model_id` was unset.

These are now fixed in v2.

This is exactly why keeping v2 isolated from v1.4 was the correct move.

## Where v2 is genuinely better right now

1. Adapter model
   - v2 has a cleaner, more portable abstraction boundary.

2. Auditability
   - worker state and transitions are more explicit.

3. Extensibility
   - adding new worker categories is conceptually cleaner in v2.

4. Comparative safety
   - v2 can evolve without destabilizing the working v1.4 branch.

## Where v1.4 still has the advantage

1. Maturity on the original use case
   - v1.4 is the known-good baseline for the original OpenCode/Codex workflow.

2. Lower conceptual overhead
   - fewer abstractions can mean easier debugging for the narrow original scope.

3. Proven behavior under the original assumptions
   - v2 still needs more realistic runs against the workflows that actually matter.

## Unknowns still blocking promotion

- real Docker adapter validation
- richer HTTP validation beyond one blocking fixture
- side-by-side execution of the same real task through v1.4 and v2
- whether v2 preserves the parts of v1.4 that mattered most in practice: respawn logic, provider balancing, and original workflow speed

## Promotion rule

Promote v2 only if all of the following become true:
- v2 continues passing local e2e
- v2 demonstrates parity or better behavior on real workflows that matter
- no critical regression is found in observability or orchestration reliability
- the migration path back to the original repo is documented clearly
