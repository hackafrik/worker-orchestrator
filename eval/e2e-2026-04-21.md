## Key Findings
1. **Worker isolation works:** opencode correctly ran in `/tmp/worker-a` without escaping.
2. **Skill auto-loading works:** `worker-orchestrator` skill was discovered and evaluated.
3. **Prompt-file pattern is correct:** No shell-quoting failures observed.
4. **Provider rate-limiting is the critical failure mode:** When API limits hit, workers hang indefinitely (no graceful degradation).
5. **Monitoring gap:** `execute_code` + `subprocess.Popen` gives no live visibility — we only see results after timeout/kill.

## Recommendations
- Add a provider-circuit-breaker: if 429 detected, respawn on alternate provider immediately.
- Shorten OpenCode `run` timeout to 60-90s when API limits are known tight.
- Pre-warm provider quota check before spawning batches.
- Consider local-model fallback (llama.cpp/ollama) for worker tasks to avoid external API limits.
